# スキルクリエイター

新しいスキルを作成し、反復的に改善するためのスキルです。

大まかには、スキル作成のプロセスは以下の通りです:

- スキルが何をすべきか、大まかにどう実現するかを決定する
- スキルのドラフトを書く
- テストプロンプトを数個作成し、スキルにアクセスできるClaudeで実行する
- ユーザーによる定性的・定量的な結果評価を支援する
  - テスト実行中に、定量的な評価基準がなければドラフトする（既にある場合はそのまま使うか、変更が必要と感じたら修正する）。そしてユーザーに説明する
  - `eval-viewer/generate_review.py` スクリプトを使って結果をユーザーに表示し、定量的な指標も確認してもらう
- ユーザーの評価結果のフィードバックに基づいてスキルを書き直す（定量的ベンチマークで明らかな欠陥がある場合も含む）
- 満足するまで繰り返す
- テストセットを拡大し、より大きな規模で再度試す

このスキルを使用する際の役割は、ユーザーがこのプロセスのどこにいるかを把握し、そこから次のステージへの進行を支援することです。例えば「Xのスキルを作りたい」と言われたら、意図を絞り込み、ドラフトを書き、テストケースを作成し、評価方法を決め、すべてのプロンプトを実行し、繰り返す手助けをします。

一方、スキルのドラフトが既にある場合は、直接eval/反復のループに入ることができます。

もちろん柔軟に対応し、ユーザーが「大量の評価は不要、感覚的にやろう」と言えば、それに従います。

スキルが完成した後は（ただし順序は柔軟）、スキルのdescription最適化ツールも実行できます。これにはスキルのトリガー精度を向上させる専用スクリプトがあります。

## ユーザーとのコミュニケーション

スキルクリエイターは、コーディング用語への精通度が大きく異なるユーザーに使われる可能性があります。最近のトレンドとして、Claudeの力に触発されて配管工がターミナルを開いたり、祖父母が「npmのインストール方法」を検索したりするようになっています。一方、大多数のユーザーはかなりコンピュータリテラシーが高いでしょう。

そこで、コミュニケーションの表現をどうするかは文脈の手がかりに注意してください。デフォルトの場合の目安:

- 「評価（evaluation）」や「ベンチマーク」は境界線上だが問題ない
- 「JSON」や「アサーション」については、ユーザーがそれらを知っている明確な手がかりが見られるまで、説明なしに使わない

疑わしい場合は用語を簡潔に説明しても構いません。

## スキルの作成

### 意図のキャプチャ

ユーザーの意図を理解することから始める。現在の会話に、ユーザーがキャプチャしたいワークフローが既に含まれている場合がある（例:「これをスキルにして」と言う場合）。その場合は、会話履歴から回答を抽出する — 使用されたツール、ステップの順序、ユーザーが行った修正、観察された入出力フォーマット。ユーザーがギャップを埋める必要があるかもしれないし、次のステップに進む前に確認が必要。

1. このスキルでClaudeに何ができるようになるべきか？
2. いつトリガーされるべきか？（どのようなユーザーフレーズ/コンテキスト）
3. 期待される出力フォーマットは？
4. スキルの動作検証のためにテストケースを設定すべきか？客観的に検証可能な出力を持つスキル（ファイル変換、データ抽出、コード生成、固定ワークフローステップ）はテストケースの恩恵を受ける。主観的な出力を持つスキル（文体、アート）は通常テストケースを必要としない。スキルの種類に基づいて適切なデフォルトを提案し、ユーザーに決定してもらう。

### インタビューとリサーチ

エッジケース、入出力フォーマット、サンプルファイル、成功基準、依存関係について積極的に質問する。テストプロンプトの作成は、この部分が固まるまで待つ。

利用可能なMCPを確認する — リサーチに有用な場合（ドキュメント検索、類似スキルの発見、ベストプラクティスの調査）、サブエージェントが利用可能ならそれを通じて並列にリサーチし、そうでなければインラインで行う。ユーザーの負担を軽減するため、コンテキストを事前に準備する。

### SKILL.md の作成

ユーザーインタビューに基づいて、以下のコンポーネントを記入する:

- **name**: スキル識別子
- **description**: トリガーのタイミングと機能。これが主要なトリガーメカニズム — スキルが何をするかと、いつ使うかの具体的なコンテキストの両方を含める。「いつ使うか」の情報はすべてここに記載し、本文には記載しない。注: 現在、Claudeはスキルを「過少トリガー」する傾向がある — 有用な場合でも使わない傾向。これに対抗するため、descriptionは少し「積極的」にする。例えば「シンプルで高速なダッシュボードの構築方法」ではなく、「シンプルで高速なダッシュボードの構築方法。ユーザーがダッシュボード、データ可視化、内部メトリクス、またはあらゆる種類のデータ表示を言及した場合は、明示的に"ダッシュボード"を要求しなくても必ずこのスキルを使用すること」のように書く
- **compatibility**: 必要なツール、依存関係（オプション、めったに不要）
- **スキルの残りの部分 :)**

### スキル作成ガイド

#### スキルの構造

```
skill-name/
├── SKILL.md（必須）
│   ├── YAMLフロントマター（name, description 必須）
│   └── Markdown指示
└── バンドルリソース（オプション）
    ├── scripts/    - 決定論的/反復的タスク用の実行可能コード
    ├── references/ - 必要に応じてコンテキストにロードされるドキュメント
    └── assets/     - 出力で使用されるファイル（テンプレート、アイコン、フォント）
```

#### プログレッシブディスクロージャー

スキルは3レベルのローディングシステムを使用する:
1. **メタデータ**（name + description）- 常にコンテキストに存在（〜100ワード）
2. **SKILL.md本文** - スキルがトリガーされるとコンテキストに入る（500行以下が理想）
3. **バンドルリソース** - 必要に応じて（無制限、スクリプトはロードせずに実行可能）

これらのワード数は目安であり、必要に応じてそれ以上にしても構わない。

**主要パターン:**
- SKILL.mdは500行以下に保つ。この制限に近づいた場合は、追加の階層と、スキルを使用するモデルがフォローアップすべき場所への明確なポインタを追加する
- SKILL.mdから参照ファイルを明確に参照し、いつ読むかのガイダンスを付ける
- 大きな参照ファイル（300行以上）には目次を含める

**ドメインの構成**: スキルが複数のドメイン/フレームワークをサポートする場合、バリアントごとに整理する:
```
cloud-deploy/
├── SKILL.md（ワークフロー + 選択）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```
Claudeは関連する参照ファイルのみを読む。

#### 驚きのない原則

言うまでもないが、スキルにはマルウェア、エクスプロイトコード、システムセキュリティを危険にさらす可能性のあるコンテンツを含めてはならない。スキルの内容は、説明された場合にユーザーを驚かせるような意図を持つべきではない。誤解を招くスキルや、不正アクセス、データ流出、その他の悪意ある活動を目的としたスキルの作成には応じない。ただし「XYZとしてロールプレイ」のようなものは問題ない。

#### 記述パターン

指示では命令形を使用する。

**出力フォーマットの定義** — 以下のように記述できる:
```markdown
## レポート構造
必ず以下のテンプレートを使用すること:
# [タイトル]
## エグゼクティブサマリー
## 主な発見事項
## 推奨事項
```

**サンプルパターン** — サンプルの掲載は有用。以下のようにフォーマットできる（「入力」と「出力」がサンプルに含まれる場合は少し変えた方がよい）:
```markdown
## コミットメッセージのフォーマット
**サンプル1:**
入力: JWTトークンによるユーザー認証を追加
出力: feat(auth): JWT ベースの認証を実装
```

### 記述スタイル

厳格なMUSTの羅列よりも、モデルに対して「なぜ」それが重要かを説明するよう心がける。心の理論を活用し、スキルを一般的で、特定のサンプルに過度に狭まらないようにする。まずドラフトを書き、新鮮な目で見直して改善する。

### テストケース

スキルのドラフトを書いた後、2〜3個の現実的なテストプロンプトを考案する — 実際のユーザーが本当に言いそうなもの。ユーザーと共有する:「いくつかテストケースを試したいと思います。これらは適切ですか？追加したいものはありますか？」そして実行する。

テストケースは `evals/evals.json` に保存する。まだアサーションは書かない — プロンプトだけ。アサーションは実行中の次のステップでドラフトする。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "ユーザーのタスクプロンプト",
      "expected_output": "期待される結果の説明",
      "files": []
    }
  ]
}
```

完全なスキーマ（後で追加する `assertions` フィールドを含む）については `references/schemas.md` を参照。

## テストケースの実行と評価

このセクションは連続したシーケンスであり、途中で止めないこと。`/skill-test` やその他のテストスキルは使用しない。

結果はスキルディレクトリの兄弟として `<skill-name>-workspace/` に配置する。ワークスペース内では、結果をイテレーション別（`iteration-1/`、`iteration-2/` など）に整理し、各テストケースにはディレクトリ（`eval-0/`、`eval-1/` など）を割り当てる。すべてを事前に作成する必要はない — 進行に応じてディレクトリを作成する。

### ステップ1: すべての実行（with-skill AND ベースライン）を同じターンで起動する

各テストケースについて、同じターンで2つのサブエージェントを起動する — 1つはスキルあり、1つはなし。重要: with-skill実行を先に起動してからベースラインに戻るのではなく、すべてを一度に起動して、ほぼ同時に完了するようにする。

**スキルあり実行:**

```
このタスクを実行:
- スキルパス: <path-to-skill>
- タスク: <eval プロンプト>
- 入力ファイル: <eval ファイル（あれば）、または "なし">
- 出力の保存先: <workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 保存する出力: <ユーザーが関心のあるもの — 例: "docxファイル"、"最終CSV">
```

**ベースライン実行**（同じプロンプトだが、ベースラインはコンテキストに依存する）:
- **新規スキルの作成**: スキルなし。同じプロンプト、スキルパスなし、`without_skill/outputs/` に保存。
- **既存スキルの改善**: 旧バージョン。編集前にスキルのスナップショットを取得（`cp -r <skill-path> <workspace>/skill-snapshot/`）、ベースラインのサブエージェントにスナップショットを指示する。`old_skill/outputs/` に保存。

各テストケースに `eval_metadata.json` を書く（アサーションはまだ空でよい）。各evalにテスト内容に基づく説明的な名前を付ける — 単なる「eval-0」ではなく。ディレクトリ名にもこの名前を使用する。このイテレーションで新しいまたは変更されたevalプロンプトを使用する場合、各新しいevalディレクトリにこれらのファイルを作成する — 前のイテレーションから引き継がれるとは想定しない。

```json
{
  "eval_id": 0,
  "eval_name": "ここに説明的な名前",
  "prompt": "ユーザーのタスクプロンプト",
  "assertions": []
}
```

### ステップ2: 実行中にアサーションをドラフトする

実行完了を待つだけでなく、この時間を有効活用する。各テストケースの定量的アサーションをドラフトし、ユーザーに説明する。アサーションが既に `evals/evals.json` にある場合は、それらをレビューしてチェック内容を説明する。

良いアサーションは客観的に検証可能で、説明的な名前を持つ — ベンチマークビューアで結果を一目見ただけで各アサーションが何をチェックしているかすぐに理解できるようにする。主観的なスキル（文体、デザイン品質）は定性的に評価した方がよい — 人間の判断が必要なものに無理にアサーションを当てはめない。

ドラフトが完了したら `eval_metadata.json` ファイルと `evals/evals.json` をアサーションで更新する。また、ビューアでユーザーが見るもの — 定性的な出力と定量的なベンチマークの両方 — を説明する。

### ステップ3: 実行完了時にタイミングデータをキャプチャする

各サブエージェントタスクが完了すると、`total_tokens` と `duration_ms` を含む通知を受け取る。このデータを実行ディレクトリの `timing.json` に即座に保存する:

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

このデータをキャプチャできる唯一の機会 — タスク通知を通じて提供され、他の場所には永続化されない。バッチ処理ではなく、到着した各通知をその場で処理する。

### ステップ4: 採点、集約、ビューアの起動

すべての実行が完了したら:

1. **各実行を採点する** — `agents/grader.md` を読んで各アサーションを出力に対して評価するグレーダーサブエージェントを起動する（またはインラインで採点する）。結果を各実行ディレクトリの `grading.json` に保存する。grading.jsonのexpectations配列は `text`、`passed`、`evidence` フィールドを使用すること（`name`/`met`/`details` や他のバリアントではなく）— ビューアはこれらの正確なフィールド名に依存する。プログラム的にチェックできるアサーションについては、目視確認ではなくスクリプトを作成して実行する — スクリプトの方が高速で、信頼性が高く、イテレーション間で再利用できる。

2. **ベンチマークに集約する** — skill-creatorディレクトリから集約スクリプトを実行する:
   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```
   これにより、各設定の pass_rate、time、tokens の mean ± stddev とデルタを含む `benchmark.json` と `benchmark.md` が生成される。benchmark.jsonを手動で生成する場合は、ビューアが期待する正確なスキーマについて `references/schemas.md` を参照。各with_skillバージョンをそのベースライン対応物の前に配置する。

3. **アナリストパスを行う** — ベンチマークデータを読み、集約統計が隠している可能性のあるパターンを表出させる。何を探すかについては `agents/analyzer.md`（「ベンチマーク結果の分析」セクション）を参照 — スキルに関係なく常にパスするアサーション（非識別的）、高分散のeval（おそらく不安定）、時間/トークンのトレードオフなど。

4. **ビューアを起動する** — 定性的な出力と定量的なデータの両方を含めて:
   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```
   イテレーション2以降では、`--previous-workspace <workspace>/iteration-<N-1>` も渡す。

   **Cowork / ヘッドレス環境:** `webbrowser.open()` が利用できない場合やディスプレイがない環境では、`--static <output_path>` を使用してサーバーを起動する代わりにスタンドアロンHTMLファイルを書き出す。フィードバックは、ユーザーが「Submit All Reviews」をクリックすると `feedback.json` ファイルとしてダウンロードされる。ダウンロード後、次のイテレーションが取得できるようワークスペースディレクトリに `feedback.json` をコピーする。

5. **ユーザーに伝える** — 例: 「ブラウザで結果を開きました。2つのタブがあります — 『Outputs』では各テストケースをクリックしてフィードバックを残せます。『Benchmark』では定量的な比較が表示されます。確認が終わったら戻ってきてください。」

### ユーザーがビューアで見るもの

「Outputs」タブは一度に1つのテストケースを表示する:
- **プロンプト**: 与えられたタスク
- **出力**: スキルが生成したファイル（可能な場合はインラインでレンダリング）
- **前回の出力**（イテレーション2以降）: 前のイテレーションの出力を折りたたみセクションで表示
- **正式な採点**（採点実行済みの場合）: アサーションのパス/失敗を折りたたみセクションで表示
- **フィードバック**: 入力すると自動保存されるテキストボックス
- **前回のフィードバック**（イテレーション2以降）: 前回のコメントをテキストボックスの下に表示

「Benchmark」タブは統計サマリーを表示する: 各設定のパスレート、タイミング、トークン使用量（eval別の内訳とアナリストの所見を含む）。

ナビゲーションは前/次ボタンまたは矢印キー。完了時に「Submit All Reviews」をクリックすると、すべてのフィードバックが `feedback.json` に保存される。

### ステップ5: フィードバックを読む

ユーザーが完了を告げたら、`feedback.json` を読む:

```json
{
  "reviews": [
    {"run_id": "eval-0-with_skill", "feedback": "チャートに軸ラベルがない", "timestamp": "..."},
    {"run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..."},
    {"run_id": "eval-2-with_skill", "feedback": "完璧、素晴らしい", "timestamp": "..."}
  ],
  "status": "complete"
}
```

空のフィードバックはユーザーが問題ないと思ったことを意味する。具体的な指摘があったテストケースに改善を集中させる。

作業が終わったらビューアサーバーを停止する:

```bash
kill $VIEWER_PID 2>/dev/null
```

## スキルの改善

これがループの核心。テストケースを実行し、ユーザーが結果をレビューし、フィードバックに基づいてスキルを改善する必要がある。

### 改善の考え方

1. **フィードバックから一般化する。** ここで起きていることの大局的なポイントは、何百万回も（文字通り、あるいはそれ以上）さまざまなプロンプトで使えるスキルを作ろうとしていること。ここではユーザーとともに少数のサンプルを何度も繰り返しているが、それは速く進むため。ユーザーはこれらのサンプルを熟知しており、新しい出力を素早く評価できる。しかし、共同開発しているスキルがそれらのサンプルでしか機能しないなら無意味。些末でオーバーフィットした変更や、圧倒的に制約的なMUSTを入れるのではなく、頑固な問題がある場合は、異なるメタファーを使ったり、異なる作業パターンを推奨したりすることを試みる。試すコストは比較的低く、素晴らしいものに出会えるかもしれない。

2. **プロンプトをスリムに保つ。** 効果を発揮していないものは削除する。最終出力だけでなくトランスクリプトも読むことを確認する — スキルがモデルに非生産的なことに大量の時間を使わせているように見える場合、スキルのその部分を削除して結果を見ることができる。

3. **理由を説明する。** モデルに求めるすべてのことの**理由**を説明するよう努力する。現在のLLMは*賢い*。良い心の理論を持ち、適切なハーネスが与えられれば、機械的な指示を超えて本当に成果を出せる。ユーザーからのフィードバックが簡潔や苛立ったものであっても、タスクを実際に理解し、ユーザーが書いた理由と内容を理解し、その理解を指示に転写する。自分がALWAYSやNEVERを大文字で書いたり、超厳格な構造を使っていることに気づいたら、それは黄色信号 — 可能であれば、リフレームしてモデルが重要性を理解するよう推論を説明する。その方がより人道的で、強力で、効果的なアプローチ。

4. **テストケース間の繰り返し作業を探す。** テスト実行のトランスクリプトを読み、すべてのサブエージェントが独立して類似のヘルパースクリプトを書いたり、同じマルチステップアプローチを取っていないか確認する。3つのテストケースすべてがサブエージェントによる `create_docx.py` や `build_chart.py` の作成を結果としていたなら、それはスキルにそのスクリプトをバンドルすべき強いシグナル。一度書いて `scripts/` に置き、スキルにそれを使うよう指示する。これにより、将来のすべての呼び出しが車輪の再発明をせずに済む。

このタスクはかなり重要で（年間数十億ドル規模の経済的価値を創出しようとしている！）、思考時間はボトルネックではない。じっくり考える。ドラフトの改訂を書き、新鮮な目で見直して改善することを勧める。ユーザーの立場に立って、何を望み、何を必要としているかを本当に理解するよう全力を尽くす。

### 反復ループ

スキルを改善した後:

1. スキルに改善を適用する
2. すべてのテストケースを新しい `iteration-<N+1>/` ディレクトリで再実行する（ベースライン実行を含む）。新規スキル作成の場合、ベースラインは常に `without_skill`（スキルなし）— イテレーション間で変わらない。既存スキルの改善の場合、ベースラインとして何が適切かは判断による: ユーザーが最初に持ち込んだ元のバージョンか、前のイテレーションか。
3. `--previous-workspace` で前のイテレーションを指してビューアを起動する
4. ユーザーがレビューして完了を告げるのを待つ
5. 新しいフィードバックを読み、再度改善、繰り返し

以下の場合まで続ける:
- ユーザーが満足だと言う
- フィードバックがすべて空（すべて問題なし）
- 有意な進展がなくなった

## 上級: ブラインド比較

スキルの2つのバージョン間でより厳密な比較を行いたい場合（例: ユーザーが「新バージョンは本当に良くなったか？」と聞く場合）、ブラインド比較システムがある。詳細は `agents/comparator.md` と `agents/analyzer.md` を参照。基本的なアイデア: 2つの出力をどちらがどちらか伝えずに独立したエージェントに渡し、品質を判断させる。次に勝者が勝った理由を分析する。

これはオプションで、サブエージェントが必要。ほとんどのユーザーには不要。人間のレビューループで通常は十分。

## Description最適化

SKILL.mdフロントマターのdescriptionフィールドは、Claudeがスキルを呼び出すかどうかを決定する主要なメカニズム。スキルの作成または改善後、トリガー精度向上のためにdescriptionの最適化を提案する。

### ステップ1: トリガーeval クエリを生成する

20個のevalクエリを作成する — トリガーすべきものとすべきでないものの混合。JSONで保存する:

```json
[
  {"query": "ユーザープロンプト", "should_trigger": true},
  {"query": "別のプロンプト", "should_trigger": false}
]
```

クエリは現実的で、Claude CodeやClaude.aiユーザーが実際に入力するものであること。抽象的なリクエストではなく、具体的で詳細なもの。例えば、ファイルパス、ユーザーの仕事や状況に関する個人的なコンテキスト、列名と値、企業名、URL。少しの背景情報。小文字や略語、タイプミス、カジュアルな口調のものも含める。長さは混合し、明確なケースよりもエッジケースに焦点を当てる（ユーザーが最終確認する機会がある）。

悪い例: `"このデータをフォーマットして"`, `"PDFからテキストを抽出"`, `"チャートを作成"`

良い例: `"上司からxlsxファイルが送られてきました（ダウンロードフォルダにあって、'Q4売上最終版FINAL v2.xlsx'みたいな名前です）。利益率をパーセンテージで表示する列を追加したいそうです。売上はC列でコストはD列だと思います"`

**トリガーすべきクエリ**（8〜10個）にはカバレッジを考慮する。同じ意図の異なる表現 — フォーマルなものとカジュアルなもの。ユーザーが明示的にスキルやファイルタイプを名前で指定しないが、明らかに必要としているケース。珍しいユースケースや、このスキルが別のスキルと競合するが勝つべきケース。

**トリガーすべきでないクエリ**（8〜10個）で最も価値があるのはニアミス — スキルとキーワードやコンセプトを共有するが、実際には異なるものが必要なクエリ。隣接ドメイン、素朴なキーワードマッチではトリガーするが実際にはすべきでない曖昧な表現、スキルが扱うものに触れているが別のツールの方が適切なコンテキストのケース。

避けるべきこと: トリガーすべきでないクエリを明らかに無関係にしない。PDFスキルの否定テストとして「フィボナッチ関数を書いて」は簡単すぎる — 何もテストしない。否定ケースは真に判断が難しいものであるべき。

### ステップ2: ユーザーとのレビュー

HTMLテンプレートを使用してevalセットをユーザーに提示する:

1. `assets/eval_review.html` からテンプレートを読む
2. プレースホルダーを置換する:
   - `__EVAL_DATA_PLACEHOLDER__` → evalアイテムのJSON配列（引用符なし — JS変数の代入）
   - `__SKILL_NAME_PLACEHOLDER__` → スキル名
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → スキルの現在のdescription
3. 一時ファイルに書き出し（例: `/tmp/eval_review_<skill-name>.html`）、開く: `open /tmp/eval_review_<skill-name>.html`
4. ユーザーはクエリの編集、should-triggerのトグル、エントリの追加/削除が可能。完了したら「Export Eval Set」をクリック
5. ファイルは `~/Downloads/eval_set.json` にダウンロードされる — 複数ある場合（例: `eval_set (1).json`）はDownloadsフォルダの最新版を確認

このステップは重要 — 悪いevalクエリは悪いdescriptionにつながる。

### ステップ3: 最適化ループを実行する

ユーザーに伝える:「これには少し時間がかかります — バックグラウンドで最適化ループを実行し、定期的にチェックします。」

evalセットをワークスペースに保存し、バックグラウンドで実行する:

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

セッションを動かしているモデルIDを使用する（現在のセッションに使われているもの）。トリガーテストがユーザーが実際に体験するものと一致するようにする。

実行中、定期的に出力をtailして、どのイテレーションにいるかやスコアの状況をユーザーに伝える。

これは完全な最適化ループを自動的に処理する。evalセットを60%のトレーニングと40%のホールドアウトテストに分割し、現在のdescriptionを評価し（各クエリを3回実行して信頼性の高いトリガー率を取得）、次に拡張思考を伴うClaudeを呼び出して失敗に基づく改善を提案する。新しい各descriptionをトレーニングとテストの両方で再評価し、最大5回反復する。完了すると、イテレーションごとの結果を示すHTMLレポートをブラウザで開き、`best_description` を含むJSONを返す — オーバーフィットを避けるため、トレーニングスコアではなくテストスコアで選択される。

### スキルトリガーの仕組み

トリガーメカニズムを理解するとより良いevalクエリを設計できる。スキルはClaudeの `available_skills` リストに名前 + descriptionで表示され、Claudeはそのdescriptionに基づいてスキルを参照するか決定する。重要なのは、Claudeは自分で簡単に処理できるタスクにはスキルを参照しないということ — 「このPDFを読んで」のような単純なワンステップクエリは、descriptionが完璧に一致しても、Claudeが基本ツールで直接処理できるため、スキルをトリガーしない場合がある。複雑な、マルチステップの、または専門的なクエリは、descriptionが一致する場合に確実にスキルをトリガーする。

つまり、evalクエリはClaudeが実際にスキルを参照することで恩恵を受けるほど実質的でなければならない。「ファイルXを読んで」のような単純なクエリはテストケースとして不適切 — descriptionの品質に関係なくスキルをトリガーしない。

### ステップ4: 結果の適用

JSON出力から `best_description` を取得し、スキルのSKILL.mdフロントマターを更新する。ユーザーにビフォー/アフターを見せ、スコアを報告する。

### パッケージ化と提示（`present_files` ツールが利用可能な場合のみ）

`present_files` ツールへのアクセスがあるか確認する。なければこのステップはスキップ。あれば、スキルをパッケージ化して.skillファイルをユーザーに提示する:

```bash
python -m scripts.package_skill <path/to/skill-folder>
```

パッケージ化後、ユーザーをインストール可能な `.skill` ファイルパスに案内する。

## Claude.ai固有の手順

Claude.aiでは、コアワークフロー（ドラフト→テスト→レビュー→改善→繰り返し）は同じだが、サブエージェントがないため一部のメカニクスが変わる。適応すべき点:

**テストケースの実行**: サブエージェントがないため並列実行不可。各テストケースについて、スキルのSKILL.mdを読み、その指示に従ってテストプロンプトを自分で実行する。一度に1つずつ。独立したサブエージェントよりも厳密さは劣る（スキルを書いた自分が実行するため、完全なコンテキストがある）が、有用なサニティチェックになる — 人間のレビューステップが補完する。ベースライン実行はスキップ — スキルを使ってタスクを完了するだけ。

**結果のレビュー**: ブラウザを開けない場合（例: Claude.aiのVMにディスプレイがない、リモートサーバー上）、ブラウザレビューアは完全にスキップする。代わりに、会話内で直接結果を提示する。各テストケースについて、プロンプトと出力を表示する。ユーザーが確認する必要のあるファイル出力（.docxや.xlsxなど）の場合は、ファイルシステムに保存してダウンロードできる場所を伝える。フィードバックをインラインで求める: 「これはどうですか？変更したいことはありますか？」

**ベンチマーク**: 定量的ベンチマークはスキップ — サブエージェントなしでのベースライン比較は意味がない。ユーザーからの定性的フィードバックに集中する。

**反復ループ**: 前と同じ — スキルを改善、テストケースを再実行、フィードバックを求める — ただしブラウザレビューアなし。ファイルシステムがあれば、結果をイテレーションディレクトリに整理できる。

**Description最適化**: このセクションは `claude` CLIツール（特に `claude -p`）が必要で、Claude Codeでのみ利用可能。Claude.aiではスキップ。

**ブラインド比較**: サブエージェントが必要。スキップ。

**パッケージ化**: `package_skill.py` スクリプトはPythonとファイルシステムがあればどこでも動作する。Claude.aiでも実行でき、ユーザーは結果の `.skill` ファイルをダウンロードできる。

## Cowork固有の手順

Coworkにいる場合、知っておくべき主なこと:

- サブエージェントがあるので、メインワークフロー（テストケースの並列起動、ベースライン実行、採点など）はすべて機能する。（ただし、タイムアウトの深刻な問題が発生した場合は、テストプロンプトを並列ではなく直列で実行しても構わない。）
- ブラウザやディスプレイがないので、evalビューアを生成する際は `--static <output_path>` を使用してサーバーを起動する代わりにスタンドアロンHTMLファイルを書き出す。ユーザーがクリックしてブラウザでHTMLを開けるリンクを提供する。
- Coworkのセットアップは、テスト実行後にClaudeがevalビューアを生成することを避ける傾向があるようなので、再度強調: CoworkでもClaude Codeでも、テスト実行後は常に人間がサンプルを見れるようevalビューアを生成し、自分でスキルを修正する前に `generate_review.py` を使用する（独自のHTMLコードを書かない）。大文字で言わせてもらう: 自分で入力を評価する*前に*EVALビューアを生成すること。できるだけ早く人間の目の前に出す!
- フィードバックの動作が異なる: サーバーが動いていないため、ビューアの「Submit All Reviews」ボタンは `feedback.json` をファイルとしてダウンロードする。そこから読み取れる（最初にアクセス許可を要求する必要があるかもしれない）。
- パッケージ化は動作する — `package_skill.py` はPythonとファイルシステムだけが必要。
- Description最適化（`run_loop.py` / `run_eval.py`）はサブプロセスを介して `claude -p` を使用するため、Coworkでも問題なく動作するはず。ただし、スキルが完全に完成し、ユーザーが良い状態だと同意するまで保存すること。
