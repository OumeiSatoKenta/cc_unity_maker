# ゲームIssue一括更新

既存のゲームIssueのbodyを最新の v2+ テンプレートに一括更新するスキル。

**引数:** 更新範囲（例: `/update-game-issues 001-020`、省略時は全件）

## 前提条件

- `gh` CLI がインストール済みで認証済み
- 対象のIssueがGitHub上に存在する

## リファレンスファイル

スキル実行時に必ず以下を読み込むこと（setup-game-issues 配下を参照）:

1. `setup-game-issues/references/issue-template-v2plus.md` — Issue bodyの正式テンプレート
2. `setup-game-issues/references/category-examples.md` — カテゴリ別の既存ゲーム参考例

## 実行フロー

### Step 1: 対象Issueの特定

```bash
# 全Issueの番号・タイトル・状態を取得
gh issue list --repo OumeiSatoKenta/cc_unity_maker --state all --limit 300 \
  --json number,title,state \
  --jq '.[] | "\(.number)\t\(.state)\t\(.title)"' | sort -n
```

Issueタイトルの `[{ID}]` からゲームIDを抽出し、引数の範囲でフィルタする。

### Step 2: テンプレートの読み込み

**必ず `setup-game-issues/references/issue-template-v2plus.md` を読み込む。**

このテンプレートがIssue bodyの正式な構造定義。テンプレート末尾の「記入ルール」と「削除すべき旧セクション」も参照する。

### Step 3: 既存Issue bodyの取得と解析

各Issueについて:

```bash
gh issue view {NUMBER} --repo OumeiSatoKenta/cc_unity_maker --json body,title,labels \
  | jq -r '.'
```

既存bodyから保持すべき情報を抽出する:
- ゲーム情報（ID、タイトル、カテゴリ、工数、シーン名）
- ゲーム概要
- コアループ
- 画面構成
- 操作仕様（v2形式で既にある場合）
- クリア/ゲームオーバー条件（記入済みの場合）
- スコア・評価基準（記入済みの場合）
- UI要素一覧（記入済みの場合）
- Unity実装上の注意（有用な内容がある場合のみ）

### Step 4: 2段階生成で新セクションを追加

既存情報を保持しつつ、不足セクションを2段階で生成する。
`references/category-examples.md` の該当カテゴリを参照して推論の質を上げる。

#### Stage 1: 既存情報の補完

既存bodyに不足しているセクションを、既存の情報 + カテゴリリファレンスから補完する:

- 操作仕様・ゲームフローテーブル（なければ生成）
- クリア/ゲームオーバー条件（空欄なら具体化）
- スコア・評価基準（なければ生成、コンボ/乗算要素を含む）
- UI要素一覧（なければ生成）

#### Stage 2: 深化セクションの生成

Stage 1で補完した全情報を踏まえて、より深い設計セクションを生成する:

- **難易度設計（5ステージ）**: コアメカニクスと操作仕様を元に、各ステージで何が変わるか具体的に設計。パラメータ変更だけでなく新ルール/要素を各ステージに追加
- **ゲームデザインパターン**: 6種カタログから最適なものを選択し、具体的な適用方法を記述
- **判断ポイント**: コアループの中で「どこを」「いつ」「何を犠牲にして」意思決定するかを3つ以上
- **チュートリアル情報**: ゲーム概要と操作仕様から初見プレイヤー向けの4テキストを生成

#### 削除対象

- `必要な GameObject 一覧`（プレースホルダーのみ）
- `Unity 実装上の注意`（プレースホルダーのみの場合。有用な内容はゲーム概要に統合）
- `Unity 実装方針`（プレースホルダーのみの場合）
- `Claude Code への依頼文（コピペ用）`

### Step 5: サンプルレビュー

最初の3件を更新したら、**残りを処理する前にユーザーに表示して確認を求める**。

- 情報量が異なるIssue（旧形式と新形式）を混ぜて3件選ぶ
- フィードバックがあれば生成方法を調整してから残りを処理

### Step 7: Issue更新

```bash
gh issue edit {NUMBER} --repo OumeiSatoKenta/cc_unity_maker --body "$NEW_BODY"
sleep 1  # レート制限対策
```

### Step 8: 検証

更新後、サンプル（最初・中間・最後の3件）を取得して目視確認する。

## 並列処理戦略

100件を処理する場合、サブエージェントで10件ずつ並列処理する:

1. **パイロット**: 最初の10件を更新し、ユーザーに確認を求める
2. **バッチ処理**: 承認後、残り90件を10件×9バッチで処理
3. **検証**: 全件のセクション存在チェック

各バッチはサブエージェントに以下を渡す:
- 担当するIssue番号リスト
- テンプレートファイルのパス
- 既存bodyのJSON

## レート制限対策

- Issue更新間隔: 1秒
- バッチ間隔: 3秒
- 失敗時: 最大3回リトライ（5秒待機）

## 注意事項

- 既存の有用な情報（ゲーム概要、操作仕様等）は必ず保持する
- プレースホルダーのみのセクションは削除する
- テンプレートのセクション順序を統一する
- チェックリスト（実装・動作確認）のチェック状態は保持する
