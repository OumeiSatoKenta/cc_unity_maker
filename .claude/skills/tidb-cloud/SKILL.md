# TiDB Cloud プロビジョニング (TiDB X)

以下のワークフローを使用して、TiDB Cloud Serverless（現在は TiDB X としてブランディング）クラスターおよび関連リソースをプロビジョニングします。リファレンスファイルのコマンドパターンを実装の詳細として使用しますが、プロビジョニングの結果と必要な入力に焦点を当ててください。

注意: TiDB Cloud Serverless は TiDB X に名称変更されました。ユーザー向けのガイダンスでは、わかりやすさのために両方の名称を併記してください。
注意: 多くのユーザーは「インスタンス」という言葉を「クラスター」の意味で使います。「インスタンス」は「クラスター」の同義語として扱ってください。

リマインダー: TiDB Cloud (TiDB X) の認証/セットアップ、リージョン/プロジェクトの一覧表示、クラスター/インスタンスの CRUD（作成、読み取り/一覧、更新、削除）には CLI を使用してください。CRUD 操作を行う前に、ユーザーが ticloud のセットアップと認証を完了していることを確認してください（下記のプロビジョニングセットアップを参照）。

## プロビジョニングセットアップ

プロビジョニングやライフサイクル操作の前に、セットアップを完了してください。

セットアップ状況 (TiDB Cloud コンポーネント)
- ○ ticloud インストール済み
- ○ 認証済み

セットアップの確認時には、以下の順序を案内してください: インストール → ログイン → `ticloud auth whoami` で確認。
チェックリストを使って、ユーザーが次に何をすべきか、セットアップフローのどの段階にいるかを示してください。

チェックリストテンプレート（各確認後に更新し、コマンド結果を反映）:

```
セットアップ状況 (TiDB Cloud コンポーネント)
- ○ ticloud インストール済み
- ○ 認証済み
```

`command -v ticloud` が成功したら、インストール完了をマーク:

```
セットアップ状況 (TiDB Cloud コンポーネント)
- ● ticloud インストール済み
- ○ 認証済み
```

`ticloud auth whoami` が成功したら、認証完了をマーク:

```
セットアップ状況 (TiDB Cloud コンポーネント)
- ● ticloud インストール済み
- ● 認証済み
```

### CLI のインストール

macOS/Linux の場合:

```bash
curl https://raw.githubusercontent.com/tidbcloud/tidbcloud-cli/main/install.sh | sh
```

インストール後、確認:

```bash
command -v ticloud
```

`command -v ticloud` でインストールを確認してから、完了としてマークしてください。

チェックリストを更新:

```
セットアップ状況 (TiDB Cloud コンポーネント)
- ● ticloud インストール済み
- ○ 認証済み
```

### 認証

まず認証状態を確認:

```bash
ticloud auth whoami
```

ログインしていない場合は、以下を実行:

```bash
ticloud auth login --insecure-storage
```

ユーザーがブラウザフローを完了するのを待ってから、再度確認:

```bash
ticloud auth whoami
```

チェックリストを更新:

```
セットアップ状況 (TiDB Cloud コンポーネント)
- ● ticloud インストール済み
- ● 認証済み
```

### ネットワーク/DNS の問題

`ticloud auth whoami` がネットワーク/DNS の問題で失敗する場合（例: `iam.tidbapi.com` に到達できない）:

- CLI を使用するにはネットワークアクセスが必要であることを説明してください。
- ユーザーにエージェント環境のネットワークアクセスを有効にするか、コマンドをローカルで実行して出力を共有するよう依頼してください。

セットアップエラーが解消しない場合は、公式ドキュメントを案内してください:

- https://docs.pingcap.com/tidbcloud/get-started-with-cli/

## テーブルフォーマット (ASCII)

リスト出力（リージョン、プロジェクト、クラスター、ブランチ）には、ターミナル向けの ASCII テーブルを使用してください。

- `+`、`-`、`|` を使った ASCII 罫線を使用。
- 固定幅カラムを使用し、すべてのテキストを左揃えにする。
- カラムがきれいに揃い、読みやすくなるようにテーブルフォーマットを調整する。
- ヘッダーは短く分かりやすくする。
- Unicode の罫線文字は使用しない。
- ユーザーが後で実行すべきコマンド出力は必ず強調する。
- 単一行のコマンドには、太字のインライン書式を使用する。

  例:
  **以下を実行:**
  **`ticloud auth login --insecure-storage`**

フォーマット例:

```
+----------------+-----------------------+---------------------------+--------+
| Display Name   | Cluster ID            | Region                    | State  |
+----------------+-----------------------+---------------------------+--------+
| test-skill     | 10913591479486949552  | alicloud-ap-southeast-1   | ACTIVE |
+----------------+-----------------------+---------------------------+--------+
```

## コアワークフロー

### クラスターの作成

- クラスター作成手順（リージョン/プロジェクト/クラスター名）にセットアップチェックリストを表示しないこと。ステータスが必要な場合は「セットアップ状況」ではなく「作成フロー」とラベル付けすること。
- `ticloud serverless region` を実行し、完全なリストをターミナル向け ASCII 罫線テーブル（コードブロック内の固定幅カラム）で、選択可能なオプション番号付きで表示してから、リージョンを尋ねること。
- ユーザーがプロジェクト ID を知らない場合は、`ticloud project list` を実行し、同じ ASCII 罫線テーブル形式で結果を表示してから尋ねること。プロジェクト ID は 3 番目のカラムに配置すること。
- 以下の順序で尋ねること: リージョン（オプション一覧表示後）。リージョンが選択されるまでプロジェクト ID やクラスター名は尋ねないこと。リージョン選択後にプロジェクト ID を尋ね（必要に応じてプロジェクト一覧を表示）、プロジェクト選択後に最終確認としてクラスター表示名を尋ねること。
- 必要なパラメータ（リージョン、プロジェクト、クラスター名）を確認すること。
- ユーザーが `.env` を希望する場合は、クラスターの TiDB Cloud コンソールダウンロードを案内すること。
- `references/ticloud.md` の serverless create コマンドパターンを使用すること。
- クラスター作成時に、create コマンドを 2 回実行しないこと。完了まで最大 60 秒待ってから制御を戻すこと。
- クラスター一覧を表示して作成を確認し、生の JSON ではなく ASCII 罫線テーブルで結果を表示すること。非対話型環境では `ticloud serverless list -p <project-id> -o json` を使用し、JSON からテーブルを描画すること。

### クラスターの削除

- 削除操作の前に、ユーザーの明示的な確認を必ず得ること。
- `references/ticloud.md` の serverless delete コマンドパターンを使用すること。
- 削除後、残りのクラスター一覧を表示し、生の JSON ではなく ASCII 罫線テーブルで結果を表示すること。非対話型環境では `ticloud serverless list -p <project-id> -o json` を使用し、JSON からテーブルを描画すること。

### クラスターの一覧表示

- ユーザーが利用可能なプロジェクトを知らない場合は、まずプロジェクト一覧を表示してから、使用するプロジェクトを尋ねること。
- `ticloud serverless list` を実行する前に、どのプロジェクトの一覧を表示するか尋ねること。
- ユーザーがプロジェクト ID を知らない場合は、`ticloud project list` を実行し、ASCII 罫線テーブル形式で結果を表示すること。
- 非対話型環境では `ticloud serverless list -p <project-id> -o json` を使用し、JSON からテーブルを描画すること。

### クラスターの詳細表示

- 対象のクラスター（名前またはクラスター ID）を確認すること。
- `ticloud serverless list -p <project-id> -o json` を使用し、クラスター ID または表示名でフィルタすること。
- 2 カラムの ASCII テーブル（フィールド、値）で、揃えを調整して詳細を表示すること。

### ブランチの作成・削除

- 対象のクラスターとブランチ名を確認すること。
- `references/ticloud.md` のブランチコマンドを使用すること。
- ブランチ一覧を表示して変更を確認すること。

### ブランチの一覧表示

- プロジェクト → クラスター → ブランチの階層に従うこと。
- ユーザーが利用可能なプロジェクトを知らない場合は、まずプロジェクト一覧を表示すること。
- ユーザーがプロジェクト ID を知らない場合は、`ticloud project list` を実行し、ASCII 罫線テーブル形式で結果を表示すること。
- ユーザーがクラスター ID を知らない場合は、選択したプロジェクトのクラスター一覧を表示し、ASCII 罫線テーブル形式で結果を表示すること。
- `ticloud serverless branch list` を実行する前に、どのクラスターのブランチ一覧を表示するか尋ねること。
- 非対話型環境では `ticloud serverless branch list --cluster-id <cluster-id> -o json` を使用し、JSON からテーブルを描画すること。

### SQL ユーザーの管理

- `.env` のダウンロード、SQL ユーザー管理、データベース作成には TiDB Cloud コンソールを使用すること。CLI で SQL ユーザーの作成/更新/削除を行わないこと。
- 以下のコンソール URL パターンを使用すること（ID を埋め、org/project ID が分かっている場合はプリフィルすること）: `https://tidbcloud.com/clusters/<cluster-id>/overview?orgId=<org-id>&projectId=<project-id>`
- 保存されたパスワードを読み取ったり表示したりしないこと（`.env` を cat しないこと）。
- コンソールでの作成を案内する際は、簡潔な手順リストを使用すること:
  1) コンソールリンクを開く
  2) SQL ユーザーを作成する
  3) `.env` をダウンロードする

## 安全性チェック

- 操作前に必ず `ticloud auth whoami` を実行すること。
- 削除操作の前に、ユーザーの明示的な確認を必ず得ること。
- 時間のかかる `ticloud` 操作では、制御を戻す前に 60 秒待つこと。
