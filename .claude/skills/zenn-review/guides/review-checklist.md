# Zenn記事レビューチェックリスト

## 1. フォーマット

- [ ] frontmatterが正しいか（title, emoji, type, topics, published）
- [ ] topicsが5つ以内で、Zennに存在するタグか
- [ ] slugが12〜50文字の英数字・ハイフンか
- [ ] `published: false`（下書き）になっているか
- [ ] コードブロックに言語指定があるか（```python, ```bash 等）

## 2. 透明性

- [ ] 「この記事について」セクションが記事冒頭（frontmatter直後）にあるか
- [ ] AI記載率が明記されているか
- [ ] 動作確認の有無が明記されているか

## 3. 文体・AI臭さ

以下のパターンに該当する箇所がないかチェックする。

### 避けるべき表現

| NG例 | 理由 | 改善例 |
|------|------|--------|
| 「なぜこの構成なのか — 設計判断の補足」 | セクションタイトルが堅すぎる | 「補足」 |
| 「最も堅牢な方法」「最も確実なアプローチ」 | 強すぎる断定 | 「〜のケースも拾える」「〜で済む」 |
| 「非常に重要です」「極めて有効です」 | 過剰な強調 | 具体的に何が良いかを書く |
| 「〜することが推奨されます」 | 受動態＋丁寧すぎ | 「〜するといい」「〜しておく」 |
| 「以下の通りです」「以下に示します」 | 形式的すぎる | 直接内容を書き始める |
| 「どれか1つ欠けても動きません」 | 実際にはそうでないケースも含めて断定 | 事実に基づいて個別に書く |

### 自然に読める文体の指針

- 筆者が同僚に説明する口調で書く
- 「〜します」「〜です」の連続を避け、「〜する」「〜した」体言止めも混ぜる
- 読者に語りかけるような表現を適度に入れる（「ここがポイントで」「困ったことに」等）

## 4. 内容の正確性

- [ ] 手順通りに実行すれば発生しない問題を「ハマりポイント」として書いていないか
- [ ] コード例が実際に動作するか（関数名、引数名、import先が正しいか）
- [ ] 実行結果の例が載っているか（特にコマンド実行の出力）
- [ ] 実行結果に個人情報（ホスト名、パスワード、Cookie等）が残っていないか
- [ ] 技術用語の意味が正しいか（score vs distance 等の方向性の混同に注意）
- [ ] 公式ドキュメントのURLが正しいか

## 5. コンテキストの明確さ

- [ ] 初出のファイル名・ツール名に、それが何でどこで使われるかの説明があるか
  - 例: `install-tools.sh` → 「`postCreateCommand` で実行されるスクリプト」
- [ ] ホストPC / コンテナ内 / ブラウザ、どこで操作するかが明確か
  - 特にDevContainer系の記事では混同しやすい
- [ ] コード例にプロジェクト固有の情報が含まれていないか（適切にマスクされているか）
- [ ] シリーズ記事のリンクが正しいシリーズに属しているか（別シリーズの記事を混ぜない）

## 6. 読者体験

- [ ] 「はじめに」で読者の課題 → この記事の解決策の流れになっているか
- [ ] 前提条件が明記されているか（「不要なもの」も書くと読者の心理的ハードルが下がる）
- [ ] 手順 + コード + 実行結果がセットになっているか
- [ ] 文字数が3,000〜8,000文字に収まっているか（Zenn記事の読まれやすいレンジ。`wc -m` で確認）
- [ ] Mermaid図やテーブルで視覚的に整理されているか
- [ ] `npx zenn preview` でレンダリングが崩れていないか
