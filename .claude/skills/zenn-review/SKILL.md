# Zenn記事レビュースキル

`zenn/articles/` の記事を `guides/review-checklist.md` の6軸でレビューする。

## レビューの流れ

### 1. 対象記事の特定

ユーザーに対象を確認する。指定がなければ最新の記事を対象にする。

```bash
ls -lt zenn/articles/*.md | head -5
```

### 2. チェックリストの読み込み

`guides/review-checklist.md` を読み、6軸のチェック項目を把握する。

### 3. レビュー実行

対象記事を読み、以下の6軸でチェックする。

#### 軸1: フォーマット
frontmatter、slug、topics、コードブロックの言語指定など、Zennの仕様に沿っているか。

#### 軸2: 透明性
「この記事について」（AI記載率・動作確認）が記事冒頭にあるか。

#### 軸3: 文体・AI臭さ
過剰な強調（「最も堅牢」「非常に重要」）、堅すぎるセクションタイトル、受動態の多用など、AIが書いた感じがする箇所を指摘する。**同僚に説明するような自然な口調**が基準。

#### 軸4: 内容の正確性
- 手順通りなら発生しない問題を「ハマりポイント」として書いていないか
- コード例のAPI名・引数名が正しいか
- 実行結果の例に個人情報が残っていないか
- 技術用語の意味が正しいか（score vs distance 等）

#### 軸5: コンテキストの明確さ
- 初出のファイル名に「何のファイルか」の説明があるか
- ホストPC / コンテナ内の区別が明確か
- プロジェクト固有の情報がマスクされているか
- シリーズ記事のリンクが正しいシリーズに属しているか

#### 軸6: 読者体験
- 「はじめに」で課題→解決の流れになっているか
- 手順 + コード + 実行結果がセットか
- 文字数が3,000〜8,000文字に収まっているか（`wc -m` で確認）

### 4. レビュー結果の報告

以下の形式で報告する。

```
## レビュー結果: {記事タイトル}

### 総合
{一言コメント}

### 指摘事項

#### [必須] {指摘タイトル}
- 箇所: {行番号 or セクション名}
- 内容: {具体的な問題}
- 修正案: {どう直すか}

#### [推奨] {指摘タイトル}
...
```

- **[必須]**: 公開前に直すべき（技術的な誤り、個人情報の露出、AI臭い表現）
- **[推奨]**: 直すとより良くなる（構成の改善、補足の追加）

### 5. 修正の適用

ユーザーが同意した指摘事項を記事に反映する。修正後、該当箇所を再確認する。
