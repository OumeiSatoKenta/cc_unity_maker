# Zenn記事執筆スキル

Zenn CLIプロジェクト（`zenn/` ディレクトリ）に技術記事を作成する。

## 前提

- Zenn CLIは `zenn/` ディレクトリにセットアップ済み
- 記事ファイルは `zenn/articles/{slug}.md` に配置
- `npx zenn preview`（`zenn/` 内で実行）でローカルプレビュー可能

## ワークフロー

### 1. 素材の収集

記事のテーマに関連する素材を収集する。

- `blog_memo/` — 開発作業メモ（手順、ハマりポイント、実行結果）
- `knowledge/` — セッション横断の教訓（backend.md, frontend.md, aws.md）
- `docs/` — プロダクト設計ドキュメント
- 既存の `zenn/articles/` — シリーズ記事の文脈

素材ファイルを読み、記事に使える情報（コマンド、コード例、エラーメッセージ、解決策）を特定する。

### 2. 記事のスキャフォールド

`guides/zenn-format.md` を読み、Zennのフォーマット制約を確認する。

以下を決定する:

- **slug**: 英数字・ハイフン、12〜50文字。内容を端的に表す英語
- **title**: 日本語。「メインテーマ — サブテーマ」形式が効果的
- **emoji**: 記事の内容を象徴する1文字
- **topics**: 最大5つ。Zennに既存のタグから選ぶ（`guides/zenn-format.md` の一覧を参照）

`templates/article.md` のテンプレートを元に記事ファイルを `zenn/articles/{slug}.md` に作成する。

### 3. 記事の執筆

テンプレートのプレースホルダーを埋めて本文を執筆する。

#### 執筆の原則

- **手順 + コード + 実行結果をセット**で書く。Zenn読者は手を動かす人が多い
- **再現可能性を重視**する。バージョン、前提条件、環境を明記する
- コードブロックには言語指定を付ける（```python, ```bash 等）
- 長いコードは要点を抜粋し、全体は GitHub リポジトリへのリンクで補完する
- `:::message alert` で注意点・補足を目立たせる
- Mermaid図でアーキテクチャや処理フローを可視化する

#### 「はじめに」の書き方

読者の課題から始め、この記事で何が解決するかを3行以内で伝える。

#### 「まとめ」の書き方

箇条書き3〜5項目で要点を整理する。記事を読まなかった人がここだけ読んでも価値がある内容にする。

#### AI利用と動作確認の透明性

記事の末尾（参考リンクの前）に以下の2点を必ず記載する。

1. **AI記載率**: この記事のうちAI（Claude Code等）が生成・執筆した割合を明記する。執筆前にユーザーに「この記事のAI記載率はどのくらいですか？」と確認する。
2. **動作確認**: 記事内のコマンドやコード例を筆者が実際に動作確認したかどうかを明記する。執筆前にユーザーに「コード例は動作確認済みですか？」と確認する。

### 4. シリーズ記事の管理

記事がシリーズの一部である場合:

1. 既存の `zenn/articles/` 内の関連記事を確認する
2. 新しい記事の「シリーズ記事」セクションに全記事へのリンクを追加する
3. **既存記事のシリーズリンクにも新記事を追加する**（相互リンク）
4. シリーズ名を統一する（例: 「Claude Code実践」シリーズ）

リンクの形式: `[記事タイトル](slug)` — Zenn内の記事はslugだけでリンクできる。

### 5. 最終確認

- [ ] frontmatterの全フィールドが正しいか
- [ ] slugが12〜50文字の英数字・ハイフンか
- [ ] topicsが5つ以内か
- [ ] `published: false` か
- [ ] コードブロックに言語指定があるか
- [ ] シリーズ記事の相互リンクが設定されているか
- [ ] AI記載率と動作確認ステータスが記載されているか
- [ ] `cd zenn && npx zenn preview` でプレビュー確認（http://localhost:8888）
