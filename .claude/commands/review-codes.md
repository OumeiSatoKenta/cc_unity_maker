# コードレビュー（3軸並列レビュー）

引数: レビュー対象パス（例: `/review-codes src/hooks/useCashFlow.ts`, `/review-codes src/components/`, `/review-codes changed`）

---

## ステップ1: レビュー対象の特定

### 引数の解析

引数に応じてレビュー対象ファイルを決定する:

- **ファイルパス** (`src/hooks/useCashFlow.ts`): そのファイルを対象とする
- **ディレクトリパス** (`src/hooks/`): そのディレクトリ内の全ソースファイル（`.ts`, `.tsx`, `.js`, `.jsx`）を対象とする。`Glob` ツールで一覧を取得する
- **`changed`**: `git diff --name-only HEAD` を `Bash` ツールで実行し、変更ファイルを自動検出する。ステージ済み・未ステージの両方を含める場合は `git diff --name-only HEAD && git diff --name-only --cached` を使用する

### ファイル数の確認

対象ファイルが **10ファイルを超える** 場合は、ユーザーに以下の確認を行う:

```
対象ファイルが [N] 件あります。全てレビューしますか？
- 選択肢1: 全てレビュー
- 選択肢2: 主要ファイルのみ（上位10件）
```

対象ファイルが0件の場合は、その旨を通知して終了する。

---

## ステップ2: コンテキスト収集

以下のドキュメントを `Read` ツールで読み込み、レビューの基準となるコンテキストを把握する:

1. `docs/architecture.md` — アーキテクチャ設計の基準
2. `docs/development-guidelines.md` — コーディング規約

また、`.steering/` ディレクトリに最新のステアリングファイルがあれば、その `design.md` を読み、今回の変更の設計意図を把握する。

---

## ステップ3: 3つのサブエージェントを並列起動

**重要**: 以下の3つのサブエージェントを `Agent` ツールで **同時に**（1つのメッセージ内で3つの `Agent` ツール呼び出しとして）起動すること。順番に起動してはならない。

### 軸1: 構造・アーキテクチャレビュー

```
Agent tool:
  subagent_type: "code-reviewer-structural"
  model: "sonnet"
  description: "Structural architecture review"
  prompt: |
    以下のファイルを構造・アーキテクチャの観点でレビューしてください。

    ## レビュー対象ファイル
    [対象ファイルパスのリスト]

    ## 参照ドキュメント
    - docs/architecture.md
    - docs/development-guidelines.md
    - docs/repository-structure.md
    [最新ステアリングファイルのパス（あれば）]

    エージェント定義（.claude/agents/code-reviewer-structural.md）に記載された5観点と出力フォーマットに従ってレビュー結果を報告してください。
```

### 軸2: 欠陥・セキュリティレビュー

```
Agent tool:
  subagent_type: "code-reviewer-secondary"
  model: "sonnet"
  description: "Bug and security review"
  prompt: |
    以下のファイルを欠陥・セキュリティの観点でレビューしてください。

    ## レビュー対象ファイル
    [対象ファイルパスのリスト]

    エージェント定義（.claude/agents/code-reviewer-secondary.md）に記載された5観点と出力フォーマットに従ってレビュー結果を報告してください。
    Codex MCP（mcp__codex__codex）を使用してセカンドオピニオンを取得してください。利用不可の場合は自身の分析のみで報告してください。
```

### 軸3: API・ドキュメント準拠レビュー

```
Agent tool:
  subagent_type: "code-reviewer-docs"
  model: "sonnet"
  description: "API and docs compliance review"
  prompt: |
    以下のファイルをAPI使用の正確性と公式ドキュメント準拠の観点でレビューしてください。

    ## レビュー対象ファイル
    [対象ファイルパスのリスト]

    ## 技術スタック

    エージェント定義（.claude/agents/code-reviewer-docs.md）に記載された5観点と出力フォーマットに従ってレビュー結果を報告してください。
    Context7 MCP でライブラリの公式ドキュメントを取得し、Serena MCP でコード構造を分析してください。
```

## ステップ4: 結果の統合

3つのサブエージェントの結果が揃ったら、以下の手順で統合レポートを作成する。

### 4-1. 重複排除

同一ファイル・同一行に対して複数の軸から同じ問題が報告されている場合:
- 最も厳しい優先度を採用する（`[必須]` > `[推奨]` > `[提案]`）
- 検出した軸を全て明記する（例: 「軸1,2で検出」）

### 4-2. 優先度順ソート

全ての問題を以下の順序で並べ替える:
1. `[必須]` — 修正が必要
2. `[推奨]` — 改善を推奨
3. `[提案]` — さらなる改善
4. `[質問]` — 設計意図の確認

### 4-3. 総合評価の算出

3軸の評価を総合して A/B/C/D を算出する:
- **A**: 全軸が概ね A-B で、`[必須]` が0件
- **B**: `[必須]` が1-2件以内で、重大なセキュリティ問題なし
- **C**: `[必須]` が3件以上、または改善が必要な領域が多い
- **D**: 重大なセキュリティ脆弱性やアーキテクチャ違反がある

## ステップ6: フィードバックループ

`[必須]` の問題を分析し、共通パターンがあるか確認する。

共通パターンが見つかった場合、ユーザーに以下を提案する:

```
[必須] の問題に共通パターンが見つかりました:
「[パターンの説明]」

knowledge/ に記録しますか？
- 選択肢1: knowledge/frontend.md に記録（React/UI関連）
- 選択肢2: knowledge/backend.md に記録（API/DB関連）
- 選択肢3: knowledge/general.md に記録（汎用）
- 選択肢4: 記録しない
```

ユーザーが承認した場合、該当する `knowledge/` ファイルに教訓を追記する。

---

## 注意事項

- 3つのサブエージェントは独立したコンテキストで動作するため、メインエージェントのコンテキストは消費しない
- レビューは詳細な分析のため、数分かかる場合がある
- Codex MCP や Context7 MCP が利用不可の場合でも、フォールバック処理により各軸は結果を返す
