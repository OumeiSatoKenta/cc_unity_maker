# add-feature 改善履歴

---

## v2 改善（2026-04-02 実施済み）

### 背景
Game 001〜100 を v1 で自動実装した結果、以下の課題が判明:
- 画面デザインが稚拙（単色矩形スプライト）
- ゲーム性が浅い（1ステージ、スコア単純）
- 初見で目的・操作・ゴールがわからない
- 各ゲームが1面で終わる
- TopMenuのカテゴリ選択が記憶されない
- お気に入り機能がない
- 連続作成時にコンテキストが溢れる

### 実施内容

#### Phase A: 共通インフラ新規作成
| ファイル | 内容 |
|---------|------|
| `Scripts/Common/InstructionPanel.cs` | ゲーム開始時チュートリアル（title/description/controls/goal）、PlayerPrefsで2回目スキップ |
| `Scripts/Common/StageManager.cs` | 5ステージ進行管理（StageConfig: speed/count/complexity）、OnStageChanged/OnAllStagesCleared イベント |
| `Scripts/Common/FavoriteManager.cs` | お気に入り管理（PlayerPrefsカンマ区切り）、シングルトン |
| `TopMenu/TopMenuManager.cs` | カテゴリ記憶（PlayerPrefs）+ ★お気に入りタブ追加 |
| `TopMenu/GameCardUI.cs` | ★/☆トグルボタン追加 |
| `Editor/SceneSetup/SetupTopMenu.cs` | FavoriteManager配線、カードにお気に入りボタン |

#### Phase B: add-feature.md v2 改訂
- **スプライト品質**: カテゴリ別カラーパレット7種、グラデーション・2px影・1pxアウトライン・2xアンチエイリアス必須
- **5ステージ進行**: StageManager統合必須、各ステージに新ルール追加（パラメータ変更だけでは不十分）
- **チュートリアル**: InstructionPanel（4テキスト）必須
- **ゲーム深化**: 判断ポイント・スキルカーブ・ゲームデザインパターン（6種カタログ）・コンボ/乗算スコア必須
- **セルフチェック**: 14項目に拡充（v1は5項目）
- **コンテキストクリア**: ステップ8を厳格化（連続実装禁止）

### v2 テスト結果（Game101 ChainReactor）
| 指標 | v1 (Game100) | v2 (Game101) |
|------|-------------|-------------|
| ステージ数 | 1 | 5（新ルール追加あり） |
| チュートリアル | なし | InstructionPanel |
| スプライト品質 | 単色矩形 | グラデーション+影+アウトライン |
| スコアシステム | 基本のみ | コンボ×連鎖×倍率 |
| 判断ポイント | 不明確 | 3つの具体的判断場面 |
| デザインパターン | なし | 連鎖+時間vs精度+リスクvsリワード |
| コード量 | ~3000行 | ~6500行 |

### ゲームデザインパターンカタログ（6種）
1. **時間 vs 精度トレードオフ**: 急ぐほど高スコアだがミスしやすい
2. **リソース管理**: 使うと強力だが有限（ボム、シールド等）
3. **連鎖/波及効果**: 1手が複数結果に影響する
4. **リスク vs リワード選択**: 安全路線と高リスク高リターン
5. **マルチタスク**: 複数要素を同時管理
6. **パターン認識 + 応用**: 規則性を学習し変化に対応

## v1 時代の改善アイデア（一部はv2で解決済み）

### add-feature ワークフロー改善

### 1. SceneSetup 前に Play Mode 停止を必須化
- ステップ5-4の冒頭に `mcp__coplaydev-mcp__manage_editor stop` を必須ステップとして追加
- Play Mode 中に SceneSetup を実行するとエラーになるケースが複数回発生した

### 2. git stash pop 後の安全確認
- ステップ6のコミット前に `git status` で意図しないファイル（削除済みゲームファイルなど）が復元されていないか確認する
- stash pop で削除済みゲームファイルが戻ってきてコンパイルエラーになった事例あり

### 3. Buggy Code 再発防止（ステップ3.5 セルフチェックに追加）
- `Physics2D` タグ・レイヤーを使う場合は `gameObject.name` 比較に切り替えることを明記
- 複数クラスの `Update()` が同時に走る場合は `_isActive` ガードを必須とする
- `PhysicsMaterial2D` などの動的生成リソースは `OnDestroy()` でクリーンアップ

### 4. 同カテゴリの既実装ゲームとの重複チェック
- ステップ3.5のセルフチェックに追加
- GameRegistry.json で同 `category` の実装済みゲームを確認し、コアメカニクスが重複していないか検証
- 93本実装後は「タップして選ぶ」系などが重複しやすい

### 5. 異なるカテゴリのパターン参照
- ステップ2-1で「直近1本のみ」参照しているが、実装ゲームの `category` が異なる場合は同カテゴリの既実装ゲームも1本追加参照する

## 良いゲームを作るための改善

### 6. フィードバック演出の必須化
- 現状: スプライトが単色矩形のみ、ヒット感・成功感の演出がない
- design.md テンプレートに「フィードバック演出」セクションを追加
- 最低限 `transform.localScale` のポップアニメ か `SpriteRenderer.color` フラッシュを1パターン必須にする

### 7. Issue テンプレートに難易度設計セクションを追加
- 現状: コアループ・操作・UI はあるが難易度カーブの記述がない
- `_totalPuzzles = 8`, `_maxMoves = 20` のような固定値だけでなく、段階的変化の設計を記述させる
- 追加セクション例: 「難易度設計（初期値・最大値・段階的変化の方針）」

### 8. ボタンのタップ領域サイズ基準の明文化
- モバイル向け最低サイズ: `sizeDelta` の最小値 `(150, 55)` を CLAUDE.md か add-feature に明記
- 現状は SceneSetup ごとにバラバラ
